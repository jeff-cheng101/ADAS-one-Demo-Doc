# vLLM 推論效能測試報告（0226_report1）

## 1. 測試目的

驗證 vLLM 在指定情境下是否達成下列門檻：

- 測試情境：每請求輸入 2,048 tokens、輸出 128 tokens
- 門檻 1：解碼吞吐量 >= 3,000 tokens/s
- 門檻 2：單請求平均速度 >= 10 tokens/s
- 門檻 3：系統處理能力 >= 1,400 RPM

## 2. 測試條件

### 2.1 硬體

- CPU：16 vCPUs（實機可見 64 vCPU）
- RAM：128 GB（實機可見約 503 GiB）
- GPU：NVIDIA RTX A6000 x2（每張 48 GB）

### 2.2 軟體與模型

- vLLM：0.16.0
- Python：3.12.3
- 模型：`mistralai/Mistral-7B-Instruct-v0.3`（FP16）
- 資料集：ShareGPT（`ShareGPT_V3_unfiltered_cleaned_split.json`）
- 測試工具：vLLM `bench serve`（本次環境中 `inferencemax` 套件可安裝，但無可直接執行 benchmark entrypoint，因此採 vLLM 官方 bench CLI 實測）

## 3. 測試命令（主要有效測試）

### 3.1 伺服器啟動

```bash
CUDA_VISIBLE_DEVICES=1 vllm serve mistralai/Mistral-7B-Instruct-v0.3 \
  --dtype half \
  --tensor-parallel-size 1 \
  --max-model-len 4096 \
  --gpu-memory-utilization 0.9 \
  --max-num-seqs 32 \
  --host 127.0.0.1 \
  --port 8000
```

### 3.2 壓測命令

```bash
vllm bench serve \
  --backend openai-chat \
  --base-url http://127.0.0.1:8000 \
  --endpoint /v1/chat/completions \
  --model mistralai/Mistral-7B-Instruct-v0.3 \
  --dataset-name sharegpt \
  --dataset-path /root/vllm-bench/ShareGPT_V3_unfiltered_cleaned_split.json \
  --input-len 2048 \
  --sharegpt-output-len 128 \
  --ignore-eos \
  --num-prompts 512 \
  --request-rate 24 \
  --max-concurrency 512 \
  --disable-tqdm \
  --save-result \
  --result-dir /root/vllm-bench \
  --result-filename bench_sharegpt_2048_128_c512_r24.json
```

> 註：`--ignore-eos` 用於固定輸出長度上限（128 tokens）。

## 4. 測試結果

來源檔案：`/root/vllm-bench/bench_sharegpt_2048_128_c512_r24.json`

- Successful requests：512
- Failed requests：0
- Request throughput：7.6695 req/s
- Output token throughput：981.6943 tok/s
- Total token throughput：2812.4546 tok/s
- Mean TTFT：20756.19 ms
- Mean TPOT：31.33 ms

## 5. 指標換算與門檻比對

### 5.1 解碼吞吐量（Decode Throughput）

- 實測：`981.69 tok/s`（以 output throughput 計）
- 目標：`3000 tok/s`
- 結論：**未達標**

### 5.2 單請求平均速度（端到端）

以 TTFT + TPOT 估算單請求完成 128 tokens 速度：

- 每請求時間約 = `20.756 + 127 * 0.03133 = 24.736 s`
- 單請求平均速度約 = `128 / 24.736 = 5.17 tok/s`
- 目標：`10 tok/s`
- 結論：**未達標**

### 5.3 系統處理能力（RPM）

- RPM = `request_throughput * 60 = 7.6695 * 60 = 460.17 RPM`
- 目標：`1400 RPM`
- 結論：**未達標**

## 6. 綜合結論

在本次實機條件與指定測試參數下（ShareGPT、2048/128、concurrency=512），vLLM 推論效能 **未達** 三項目標門檻。

## 7. 影響因素與觀察

1. 本機 `GPU0` 存在既有 Python/vLLM 相關進程常駐占用 VRAM（約 19 GB），對雙卡 TP=2 穩定性造成干擾。
2. 雙卡 TP=2 配置測試曾出現連線中斷或輸出 token 計數異常，故本報告採用成功率 100% 的有效 run 作為主要依據。
3. 在當前資源干擾狀態下，瓶頸明顯體現在 TTFT 偏高，進而壓低端到端 tokens/s 與 RPM。
